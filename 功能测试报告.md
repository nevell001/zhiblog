## 1. 文章管理功能测试

### 1.1 测试目标
验证博客文章的创建、编辑、发布和删除功能是否正常工作。

### 1.2 测试用例

#### 1.2.1 创建文章
**测试步骤**：
1. 登录系统，导航至文章管理页面
2. 点击「新增」按钮，打开文章编辑对话框
3. 填写文章标题、内容、选择分类和标签
4. 设置文章状态（发布/草稿）
5. 点击「确定」按钮提交

**预期结果**：
- 文章成功创建并显示在列表中
- 显示「新增成功」提示信息
- 文章状态正确显示

#### 1.2.2 编辑文章
**测试步骤**：
1. 在文章列表中选择一篇文章
2. 点击「修改」按钮，打开编辑对话框
3. 修改文章内容（标题、内容等）
4. 点击「确定」按钮保存修改

**预期结果**：
- 文章内容成功更新
- 显示「修改成功」提示信息
- 列表中显示更新后的文章信息

#### 1.2.3 发布文章
**测试步骤**：
1. 创建一篇状态为「草稿」的文章
2. 编辑该文章，将状态修改为「发布」
3. 保存修改

**预期结果**：
- 文章状态成功更新为「发布」
- 在前台博客页面可以看到该文章

#### 1.2.4 删除文章
**测试步骤**：
1. 在文章列表中选择一篇或多篇文章
2. 点击「删除」按钮
3. 在确认对话框中点击「确定」

**预期结果**：
- 文章成功从列表中移除
- 显示「删除成功」提示信息

### 1.3 测试发现的问题

1. **数据处理逻辑复杂**：在文章提交函数中，存在大量手动数据类型转换和默认值设置，增加了出错风险。
2. **错误处理冗余**：新增和修改操作中的错误处理逻辑几乎完全相同，代码重复。
3. **标签ID处理**：存在对tagIds类型的特殊处理逻辑，可能在某些情况下导致标签关联失败。
4. **封面图片URL处理**：getCoverUrl函数在构建URL时依赖环境变量，可能在配置不正确时导致图片显示问题。
5. **表单验证规则简单**：目前仅对标题和内容做了非空验证，缺乏更严格的业务规则验证。

### 1.4 改进建议

1. **简化数据处理**：使用更统一的数据处理方式，或考虑使用数据转换库来减少手动类型转换。
2. **抽象错误处理**：将错误处理逻辑抽象为单独的函数，减少代码重复。
3. **加强表单验证**：增加更多的业务规则验证，如标题长度限制、内容格式要求等。
4. **完善标签处理**：确保tagIds始终保持为正确的数组格式，避免类型转换错误。
5. **优化用户体验**：
   - 文章内容编辑器可以增加自动保存功能
   - 添加更详细的表单验证错误提示
   - 实现批量操作的进度指示
6. **代码健壮性增强**：
   - 增加更多的空值检查
   - 添加边界条件处理
   - 优化异步操作的错误处理流程

2. **加强表单验证**：

   - 添加排序号的范围验证
3. **删除安全保护**：
   - 删除前检查分类是否被文章引用
   - 提供批量删除确认和进度显示
4. **增强分类功能**：
   - 添加分类启用/禁用功能
   - 支持分类层级结构（父子分类）
   - 添加分类图标或颜色设置
5. **优化用户体验**：
   - 在列表中显示每个分类的文章数量
   - 添加分类搜索建议
   - 实现分类排序的拖拽功能

## 4. 用户权限控制测试

### 4.1 测试目标
验证系统的权限控制机制是否正常工作，确保不同角色的用户只能访问其被授权的功能模块。

### 4.2 测试用例

#### 4.2.1 超级管理员权限测试
**测试步骤**：使用admin账号登录系统，验证是否能访问所有功能模块。
**预期结果**：可以访问所有功能，包括文章管理、分类管理、标签管理、评论管理等。

#### 4.2.2 普通管理员权限测试
**测试步骤**：使用普通管理员账号登录系统，验证权限范围内的功能访问。
**预期结果**：只能访问被授权的功能模块，未授权的功能应被隐藏或访问受限。

#### 4.2.3 权限指令测试
**测试步骤**：验证`v-hasPermi`指令是否正确控制按钮显示。
**预期结果**：用户没有对应权限时，相关操作按钮应被隐藏。

#### 4.2.4 角色切换测试
**测试步骤**：修改用户角色，验证权限变化是否生效。
**预期结果**：角色修改后，用户权限应相应更新，需重新登录使权限生效。

### 4.3 测试发现的问题

1. **权限缓存问题**：角色和权限修改后需要重新登录才能生效，用户体验不佳。
2. **错误处理不完善**：权限验证失败时缺少友好提示，用户可能不理解为什么某些功能不可用。
3. **权限粒度不够细**：某些功能模块的权限控制可能过于粗放，无法满足精细化的权限管理需求。
4. **缺少权限审计**：没有记录用户的权限使用情况，不利于安全审计。
5. **Token管理简单**：Token存储在Cookie中，安全性可能不够高。

### 4.4 改进建议

1. **实现权限实时刷新**：修改用户角色或权限后，系统应能实时更新用户的权限，无需重新登录。
2. **完善权限提示机制**：当用户尝试访问无权限功能时，提供明确的提示信息。
3. **细化权限控制粒度**：支持更细粒度的权限控制，如数据级别权限。
4. **添加权限审计日志**：记录用户的权限使用情况，便于安全审计和问题排查。
5. **增强Token安全性**：考虑使用更安全的Token管理方式，如JWT Token存储在HttpOnly Cookie中。

## 5. 菜单导航功能测试

### 测试目标
验证系统所有相关菜单的显示、点击、跳转和高亮状态是否正常工作，确保用户可以方便地导航到各个功能模块。

### 测试用例
1. **菜单显示测试**
   - 测试步骤：登录系统，检查侧边栏中所有菜单的显示情况。
   - 预期结果：所有有权限的菜单应正确显示，包含图标和名称，层级结构清晰。

2. **菜单点击跳转测试**
   - 测试步骤：点击各个菜单项，验证是否能正确跳转到对应页面。
   - 预期结果：点击菜单后页面应正确跳转，URL路径与配置一致。

3. **菜单高亮状态测试**
   - 测试步骤：访问各个功能页面，验证对应菜单项是否高亮显示。
   - 预期结果：当前访问页面的菜单项应高亮显示，面包屑导航也应正确显示当前路径。

4. **权限相关菜单显示测试**
   - 测试步骤：使用不同权限的用户登录，验证菜单显示是否符合权限设置。
   - 预期结果：无权限的菜单应不显示或显示为禁用状态。

### 测试发现的问题
1. **路由匹配逻辑复杂**：路由守卫中的匹配逻辑较为复杂，可能存在潜在的问题。
2. **白名单管理简单**：白名单硬编码在路由守卫中，不易维护。
3. **缺少导航状态持久化**：刷新页面后，菜单高亮状态可能会丢失。
4. **权限验证错误处理不完善**：权限不足时只有日志输出，没有给用户明确提示。
5. **嵌套菜单层级过多**：某些功能的嵌套菜单层级较多，导航路径较长。

### 改进建议
1. **简化路由匹配逻辑**：优化路由守卫中的匹配算法，提高代码可读性和维护性。
2. **集中管理白名单**：将路由白名单抽取为单独的配置文件，便于管理和维护。
3. **实现导航状态持久化**：使用localStorage或sessionStorage保存导航状态，刷新页面后保持高亮。
4. **增强权限验证提示**：权限不足时，给用户明确的提示信息，而不仅仅是日志输出。
5. **优化菜单结构**：重新设计菜单结构，减少嵌套层级，提高导航效率。
