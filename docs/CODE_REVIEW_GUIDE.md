## 概述

代码审查（Code Review）是保证代码质量的重要环节。本文档定义了 ZhiBlog 项目的代码审查流程和标准。

## 代码规范工具

### 前端 (Vue 3)

**工具**:
- ESLint: 代码质量检查
- Prettier: 代码格式化

**使用方法**:
```bash
# 检查代码规范
npm run lint:check

# 自动修复代码规范问题
npm run lint

# 格式化代码
npm run format

# 检查代码格式
npm run format:check
```

**配置文件**:
- [.eslintrc.cjs](../ruoyi-ui/.eslintrc.cjs) - ESLint 配置
- [.prettierrc.json](../ruoyi-ui/.prettierrc.json) - Prettier 配置

### 后端 (Java 17)

**工具**:
- Checkstyle: 代码风格检查

**使用方法**:
```bash
# 检查代码规范
mvn checkstyle:check

# 自动生成报告
mvn checkstyle:checkstyle
```

**配置文件**:
- [checkstyle.xml](../checkstyle.xml) - Checkstyle 配置

## 代码审查流程

### 1. 提交前检查

在提交代码前，开发者必须：

**前端**:
```bash
# 运行代码检查
npm run lint:check

# 检查代码格式
npm run format:check

# 如果有错误，运行自动修复
npm run lint
npm run format
```

**后端**:
```bash
# 编译并检查代码规范
mvn clean compile checkstyle:check
```

### 2. 创建 Pull Request

创建 PR 时，必须包含：

- **清晰的标题**: 描述修改内容
- **详细的描述**:
  - 修改原因
  - 实现方案
  - 测试情况
  - 相关 Issue
- **关联的 Issue**: 使用 `#issue_number` 格式
- **截图/UI变更**: 如果涉及UI修改

### 3. 代码审查标准

审查者应检查：

- ✅ 功能正确性
- ✅ 代码规范（通过工具检查）
- ✅ 代码可读性
- ✅ 性能影响
- ✅ 安全问题
- ✅ 测试覆盖
- ✅ 文档完整性

### 4. 审查流程

```
开发者提交 → 自动检查 → 人工审查 → 修改反馈 → 批准合并
```

## 最佳实践

### 提交信息规范

使用约定式提交（Conventional Commits）:

```
<type>(<scope>): <subject>

<body>

<footer>
```

**类型（type）**:
- `feat`: 新功能
- `fix`: 修复bug
- `docs`: 文档更新
- `style`: 代码格式（不影响代码运行）
- `refactor`: 重构
- `perf`: 性能优化
- `test`: 测试相关
- `chore`: 构建/工具变更

**示例**:
```
feat(blog): 添加文章分类功能

- 实现文章分类CRUD接口
- 添加分类树形结构展示
- 完善分类缓存逻辑

Closes #123
```

### 代码审查原则

1. **建设性反馈**: 提出改进建议，而非批评
2. **解释原因**: 说明为什么需要修改
3. **提供示例**: 给出代码示例更好
4. **及时响应**: 尽快处理审查意见
5. **相互尊重**: 保持专业和友善

### 审查响应时间

- 开发者: 提交后 24 小时内响应反馈
- 审查者: 收到 PR 后 24 小时内完成审查

### 常见问题

**Q: 代码规范太严格怎么办？**
A: 工具配置已经过调整，遵循团队约定。如需调整，在团队讨论后修改配置。

**Q: 审查意见不一致怎么办？**
A: 邀请第三方审查者或团队成员讨论决定。

**Q: 紧急bug修复是否需要完整审查？**
A: 可以简化流程，但事后必须补充审查。
